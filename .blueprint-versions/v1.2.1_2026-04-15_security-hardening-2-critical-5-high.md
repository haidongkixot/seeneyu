The architecture, multi-agent system, tech stack, UX patterns, and build order are identical — only the **domain content** changes.

## 1. System Overview

### What seeneyu is

A **skill-coaching SaaS platform** (web + mobile) where users:
1. **Watch** curated content demonstrating a skill
2. **Observe** a structured breakdown of what makes it effective
3. **Practice** by recording themselves mimicking the skill (step-by-step or full performance)
4. **Get AI feedback** on their attempt (client-side ML scoring + optional LLM analysis)
5. **Repeat** with progressively harder content, guided by a personalized learning path

This loop is gamified (XP, streaks, badges, leaderboards) and monetized (3-tier subscription).

### What makes this architecture reusable

The system separates **domain content** (what skill is being taught) from **platform mechanics** (how learning, feedback, gamification, payments, and admin work). When building a new app:

- **Keep 100%**: Auth, admin CMS, gamification, subscriptions, payments, error logging, signal system, role system, deployment pipeline, mobile scaffold
- **Keep ~90%**: Recording flow, feedback display, library UI, practice stepper, foundation courses, discussions, AI assistant
- **Adapt ~50%**: AI scoring model (MediaPipe → breathing-specific sensors/analysis), content taxonomy (skills/difficulty), observation guides, practice steps
- **Replace 100%**: Seed data, video content, skill categories, domain-specific copy/branding

## 2. Multi-Agent Role System

seeneyu is built by **8 specialized Claude Code agents**, each running in its own terminal window with its own `CLAUDE.md` instructions. They coordinate via file-based signals.

### The 8 roles

| # | Role | Directory | What it owns | Writes code? |
|---|------|-----------|--------------|-------------|
| 1 | **PM** | `roles/pm/` | Milestones, decisions, task assignment, phase gates | No |
| 2 | **Designer** | `roles/designer/` | Design system, component specs, UI review | No (specs only) |
| 3 | **Backend Engineer** | `roles/backend-engineer/` | All production code: APIs, services, Prisma schema, pages, components | Yes |
| 4 | **Data Engineer** | `roles/data-engineer/` | Content pipelines, seed data, taxonomy, data validation | Yes (scripts only) |
| 5 | **Tester** | `roles/tester/` | Test cases, bug reports, milestone sign-off | Yes (tests only) |
| 6 | **Builder** | `roles/builder/` | Git, deployment, env vars, DB ops, mobile scaffold | Yes (config only) |
| 7 | **Reporter** | `roles/reporter/` | Activity log, shared knowledge, onboarding doc, decisions summary | No |
| 8 | **Marketer** | `roles/marketer/` | Brand brief, pitch deck, VC materials, website copy | No |

### How to set up each role

Each role has a `CLAUDE.md` file in its directory. When you `cd roles/<role>/` and open Claude Code, Claude reads that file and becomes that specialist. The CLAUDE.md must contain:

```markdown
# Role: <Name>
# <project-name> project — <absolute path>

## Your Identity
You are the <Role> for <project>. <1-sentence purpose>. <What you never do>.

## Shared data pool path
`../../.shared/` (relative to this directory)

## SESSION PROTOCOL — Do this EVERY session, in order:
### Step 1: Read the signal board
### Step 2: Read project state files
### Step 3: Do work / process signals
### Step 4: Signal other roles when done

## Your Skills
<Detailed description of each responsibility with file formats and examples>

## Output Locations
<Exact file paths for all artifacts this role produces>
```

**Key design decisions in the role system:**
- **PM is the only role that edits state files** (`milestones.json`, `project-state.json`, `decisions.json`). All other roles signal PM with updates. This prevents data corruption from concurrent edits.
- **Tester has veto power**. No milestone is complete without Tester sign-off. This forces quality.
- **Reporter is the team memory**. Any role starting a fresh session reads Reporter's `shared-knowledge.md` and `onboarding.md` to get instant context.
- **Every long-running role checkpoints progress** to JSON files so work survives context resets.

### Role interaction flow

```
User gives high-level goal to PM
    ↓
PM breaks into milestones → assigns tasks via signals
    ↓
Designer writes specs → signals Backend Engineer + Reporter
    ↓
Data Engineer curates content → signals Backend Engineer + Tester
    ↓
Backend Engineer implements → signals Tester + Reporter
    ↓
Tester reviews → signals PM (approved/blocked) + Reporter
    ↓
PM marks milestone complete → signals Builder to deploy
    ↓
Builder deploys → signals Tester (smoke test) + Reporter
    ↓
Reporter logs everything → updates shared knowledge
```

### How to replicate for your project

1. Copy `roles/` directory structure from seeneyu
2. Copy `scripts/signal-send.js` and `scripts/signal-done.js` (file-locking signal system)
3. Copy `.shared/` directory structure
4. Adapt each role's `CLAUDE.md` — change project name, domain references, skill categories
5. Initialize empty state files (`project-state.json`, `milestones.json`, `decisions.json`, `board.json`)

**Reference files in seeneyu repo:**
- `roles/pm/CLAUDE.md` — PM role definition (most complex, ~200 lines)
- `roles/designer/CLAUDE.md` — Designer with component spec format
- `roles/tester/CLAUDE.md` — Tester with bug report JSON format
- `roles/data-engineer/CLAUDE.md` — Data Engineer with checkpoint pattern
- `roles/reporter/CLAUDE.md` — Reporter with activity log format
- `roles/builder/CLAUDE.md` — Builder with deployment runbook
- `roles/backend-engineer/CLAUDE.md` — Backend Engineer with progress checkpoint
- `roles/marketer/CLAUDE.md` — Marketer with pitch deck structure
- `CLAUDE.md` (root) — Project overview read by all roles

## 3. Signal Protocol & Shared State

### Signal board (centralized)

All signals live in **one file**: `.shared/signals/board.json`. Every role reads this file at session start and filters by `"to": "<their-role>"`.

```json
{
  "_meta": { "version": 2, "open_count": 48 },
  "signals": [
    {
      "id": "sig_pm_designer_1774411655944",
      "from": "pm",
      "to": "designer",
      "type": "task-assign",
      "task": "m26-design-spec",
      "priority": "high",
      "message": "Design the registration approval flow...",
      "timestamp": "2026-03-25T04:07:35.944Z",
      "status": "unread"
    }
  ]
}
```

**Signal types**: `task-assign`, `task-complete`, `task-blocked`, `fyi`, `milestone-approved`, `deploy-ready`, `review-request`

**Signal lifecycle**: `unread` → `read` → `done` (moved to `archive.json`)

### File locking (critical for concurrency)

Both signal scripts use `O_EXCL` atomic file creation for locking:
- Lock file: `board.json.lock`
- Retry every 50ms, timeout at 10 seconds
- Stale lock detection: auto-remove locks older than 30 seconds
- Writes are atomic: write to temp file, then rename

**Reference**: `scripts/signal-send.js`, `scripts/signal-done.js`

### Shared state directory

```
.shared/
├── signals/
│   ├── board.json              ← Active signals (all roles read, scripts write)
│   └── archive.json            ← Completed signals (append-only)
├── state/                      ← ONLY PM EDITS THESE
│   ├── project-state.json      ← Current phase, active tasks, blockers, bugs
│   ├── milestones.json         ← All milestones with status + gate criteria
│   └── decisions.json          ← Architectural/product decisions log
├── memory/
│   ├── shared-knowledge.md     ← Reporter maintains: scannable project snapshot
│   ├── design-system.md        ← Designer maintains: Tailwind tokens, typography
│   └── tech-stack.md           ← All roles read: framework, deps, versions
└── outputs/
    ├── design/                 ← Designer: component specs, flow diagrams
    ├── data/                   ← Data Engineer: seed files, pipelines, checkpoints
    ├── bugs/                   ← Tester: test cases, bug reports, coverage
    ├── builder/                ← Builder: deployment state, env template
    ├── backend-engineer/       ← Backend: progress checkpoint
    ├── marketer/               ← Marketer: brand brief, pitch deck
    └── reports/                ← Reporter: activity log, milestone reports
```

### Core content model pattern (adapt this)

In seeneyu, the central model is `Clip`:

```prisma
model Clip {
  id                    String   @id @default(cuid())
  // Content source
  youtubeVideoId        String
  mediaType             String   @default("youtube")  // youtube | ai_image | ai_video
  mediaUrl              String?
  // Domain metadata (CHANGE THESE for your domain)
  movieTitle            String
  year                  Int?
  characterName         String?
  actorName             String?
  sceneDescription      String   @db.Text
  // Classification (ADAPT categories)
  skillCategory         String   // e.g., "eye-contact" → "diaphragmatic-breathing"
  difficulty            String   // beginner | intermediate | advanced
  difficultyScore       Int?     // composite score from dimensions
  // Scoring dimensions (ADAPT these 4 dimensions)
  signalClarity         Int?     // 1-3
  noiseLevel            Int?     // 1-3
  contextDependency     Int?     // 1-3
  replicationDifficulty Int?     // 1-3
  // Rich content
  annotation            String?  @db.Text
  script                String?  @db.Text
  observationGuide      Json?    // structured breakdown
  // Relations
  annotations           Annotation[]
  sessions              UserSession[]
  practiceSteps         PracticeStep[]

  @@index([skillCategory])
  @@index([difficulty])
  @@index([skillCategory, difficulty])
}
```

### Difficulty scoring formula

Score 4 domain-specific dimensions (1-3 each). Sum determines tier:
- **Sum 4-5** = Beginner
- **Sum 6-8** = Intermediate
- **Sum 9-12** = Advanced

For breathing, dimensions might be: Rhythm Complexity, Duration Demand, Body Awareness Required, Environmental Sensitivity.

```typescript
export const authOptions: NextAuthOptions = {
  adapter: PrismaAdapter(prisma),
  session: { strategy: 'jwt' },

    CredentialsProvider({
      credentials: { email: {}, password: {} },
      async authorize(credentials) {
        const user = await prisma.user.findUnique({ where: { email } })
        if (!user?.passwordHash) return null

        if (!valid) return null
        if (user.status !== 'approved') throw new Error(`status:${user.status}`)
        return { id: user.id, email: user.email, name: user.name, role: user.role }
      }
    })
  ],
  callbacks: {
    jwt({ token, user }) { /* attach id, role, status to token */ },
    session({ session, token }) { /* attach token fields to session.user */ }
  }
}
```

## 17. Critical Rules & Constraints

These rules were learned through real bugs and data loss in seeneyu. Follow them exactly:

1. **PM is source of truth** — Only PM edits `milestones.json`, `project-state.json`, `decisions.json`. All other roles signal PM with updates.

2. **No milestone complete without Tester sign-off** — Code-complete is not complete. Tester must explicitly approve.

3. **File-based state only** — Nothing lives only in Claude's context memory. Every decision, every piece of progress, every output goes to a file.

4. **Checkpoint everything** — Any pipeline or multi-step task saves progress to a JSON checkpoint file. Context resets will happen. The next session must resume, not restart.

5. **Append-only logs** — Activity log and signals are never deleted or edited. Only appended or archived.

6. **Signal system uses file locking** — Never manually edit `board.json`. Always use `signal-send.js` and `signal-done.js` to prevent race conditions.

7. **Never self-host video** — Use YouTube IFrame API or equivalent CDN. Zero hosting cost, no copyright risk.

8. **`(prisma as any)` cast pattern** — When `prisma generate` can't run locally (Windows DLL lock), use this cast. Vercel's `postinstall` hook handles regeneration on deploy.

9. **Never skip Designer review** — When Backend Engineer implements a component, Designer reviews against spec before Tester tests.

10. **Errors at system boundaries only** — Don't add error handling for internal code paths. Validate at: user input, API requests, external service calls.

11. **Admin password must be changed** — The default `changeme123` from `create-admin.ts` is for bootstrapping only.

12. **Relative dates in signals** — Always convert "Thursday" → "2026-03-28" in signal messages so they remain interpretable after time passes.

## 19. Recent Updates — v1.1.0 (2026-04-08)

This section captures the **Learning System Upgrade** sprint that shipped after the M0–M73 baseline. Snapshot of v1.0.0 lives at `seeneyu/.blueprint-versions/v1.0.0_2026-04-08_pre-learning-system.md`.

### What changed at the system level

Four parallel workstreams shipped together over 7 sprints (~13 days) with a single coordinated schema migration to avoid conflicts:

| Code | Workstream | What it does | Why it matters |
|------|-----------|-------------|----------------|
| **I1** | Session Delta | After every practice, feedback page shows score deltas vs the previous attempt on the same clip + per-dimension trajectory. Progress page shows SVG sparklines per clip. | Makes improvement *visible* — without this, users can't see they're getting better. |
| **I2** | Feedforward Best-Moment Highlight | Per-frame snapshot scores are persisted, peak 3-second window is identified, "Your Best Moment" card on feedback page shows the user their highest-quality stretch. | Strongest known Video Self-Modeling effect — watching your own peak builds skill faster than watching a model. |
| **I3** | Spaced Review Scheduling | Each completed session gets a `nextReviewAt` timestamp based on score (1d/3d/7d/21d). Dashboard surfaces "Ready to Review" cards. Activity planner adds review activities at top priority. | Closes the retention loop — without this, learners practice once and forget. |
| **I4** | Ungated Dimension Bars | Free users now see per-dimension scores on feedback (was Standard+ only). | 1-line change, 80% of free users were blocked from the deliberate-practice loop. |
| **LC** | Personalized Learning Curve | Users select genres, purpose, character traits, and gender. Content (Library, Arcade, Foundation, Games) is filtered or badged "Recommended" against tagged clips. AI auto-tags every clip with confidence scores. | Transforms a flat catalog into a curated journey per user. |

### Schema additions (single migration)

```prisma

  // I2: feedforward
  snapshotScores       Json?    // [{sec, score}] per-frame
  recordingDurationSec Int?
  // I3: spaced review
  nextReviewAt         DateTime?
  reviewCount          Int      @default(0)
```

### Critical fixes (post-deploy)

These were discovered after v1.1.0 went live and fixed in the same version cycle:

1. **MediaPipe feedback was generic and duplicate** — Root cause: `clip.observationGuide`, `clip.annotation`, and `clip.practiceSteps` existed in DB but were never passed to the GPT prompt. The feedback generator only received `{skill, character, scene}` and fell back to skill-generic templates. Fix: extended `FeedbackContext` interface, fetched practiceSteps relation in feedback route, rewrote GPT prompt to require explicit references to observation moments + step numbers + character + dimension scores. The `observationGuide.moments` array (with `atSecond`, `technique`, `what`, `why`) now serves as MediaPipe checkpoints.

2. **XP was never accumulating** — Root cause: `awardXp()` worked correctly but was never called from the high-traffic user actions (feedback submission, arcade attempts, foundation lessons). Only daily-quest completion called it. Fix: added `awardXp()` calls to feedback route (15 base + 10/20 score bonus), arcade attempts route (challenge.xpReward + score bonus), and foundation progress route (with first-time-only guard against farming).

3. **GamificationBar showed stale data** — Root cause: `useEffect([])` only fetched profile on mount, never refetched. Fix: refetch on pathname change, on `xp:awarded` window event (fired by arcade results + feedback poller), and on focus. Uses `cache: 'no-store'`.

4. **Onboarding tour blocked page interaction** — Root cause: full-screen SVG overlay had `pointer-events: auto`, capturing all clicks even outside the cutout. Fix: replaced with a `pointer-events: none` glow border (animated amber pulse + 35% surrounding dim via box-shadow). Users can now actually interact with the highlighted element.

5. **"Recommended" badges never appeared on Arcade/Games** — Root cause: client-side `fetch('/api/preferences')` didn't include cookies → 401 → empty prefs. Fix: added `credentials: 'include'`.

6. **Library `forYou` was filtering wrong** — fixed in same sprint with the matcher integration.

7. **Arcade camera too tall** — replaced `flex-1 min-h-[200px]` with `aspect-video max-h-[45vh]`.

### Pipeline pattern: AI clip generation

The `import-hollywood-clips.js` script is the reference for batch content generation. Reuse for any skill-coaching app:

1. **Curated JSON** with manual fields (movieTitle, character, scene, skill, difficulty, annotation, startSec, endSec)
2. **Skip-if-exists** by `youtubeVideoId` for resumability
3. **Per-clip pipeline**: Clip create → YouTube transcript (optional) → GPT-4o observation guide → GPT-4o practice steps → DALL-E 2 demo images → Vercel Blob upload → GPT-4o auto-tagging
4. **Resilience**: each phase wrapped in try/catch — clip creation is hard, downstream phases are soft warnings
5. **Rate limiting**: 1s between clips, 500ms between image generations
6. **Flags**: `--dry`, `--limit N`

### Architectural lessons learned (v1.1.0)

1. **Schema migrations should batch across workstreams** — we shipped 6 fields + 2 tables + 1 index in one `prisma db push`. No conflicts, no client regeneration loops.
2. **Shared files should be touched in single passes** — feedback route got I1+I2+I3 in one commit, dashboard got I3+LC in one commit, activity planner got I3+LC in one commit. No merge thrash.
3. **Existing data must always be passed to AI prompts** — we built `observationGuide` months ago but never fed it to the feedback generator. Audit your AI prompts: what data exists in DB but isn't in the prompt context?
4. **XP awarding belongs in the action route, not a side channel** — we had a perfectly working `processActivity` endpoint that nothing called. Inline `awardXp()` calls in the action routes are simpler and harder to forget.
5. **UI components need to refetch on state-changing actions** — `useEffect([])` is a footgun for any component that displays data that other actions can change. Use route changes + custom events as refresh triggers.
6. **Backward compatibility via "absent = inclusive"** — null preferences = show everything. Untagged clips = pass all filters. Null snapshotScores = hide BestMomentCard. This let us ship without breaking existing users.

### Architecture lessons learned (v1.2.0)

1. **Template-based feedback is a trap** — `generateDetailedMicroFeedback()` had ~15 hardcoded strings keyed to score thresholds. Users with similar scores (which is common — beginners cluster around the same range) got identical text every time. Always pass actual scores to GPT for text generation.

2. **Video thumbnails for AI clips require special handling** — YouTube CDN works for YouTube clips but AI clips have fake IDs (`ai-xxx`). Use `<video preload=metadata>` with auto-seek for browser rendering, or ffmpeg for server-side extraction. Never assume all clips have valid YouTube IDs.

3. **Camera cleanup needs three layers** — Component unmount alone isn't enough. Add `visibilitychange` (tab switch) and `beforeunload` (tab close) listeners. The `useCameraStream` hook encapsulates all three — use it instead of rolling your own `getUserMedia`.

4. **MediaPipe warmup must happen BEFORE recording** — The first `detectAll()` call compiles WASM + initializes the GPU, blocking the main thread for 7-8 seconds. Running it during camera initialization (while the user reads the step instructions) makes subsequent calls during recording fast.

5. **Bounded concurrency via semaphore for batch dispatches** — When pushing N video generation requests, use a simple counter-based semaphore (not a queue library) with `waitUntil()` for fire-and-forget. Default concurrency = 3, configurable 1-5. HTTP handler returns in seconds regardless of batch size.

6. **Selective publish > all-or-nothing** — Admins want to review individual items before publishing. Checkboxes + "Select All" toggle + count badge in the publish button is the right UX. The API should accept optional `requestIds[]` — if omitted, publish all.

## 21. Security Hardening — v1.2.1 (2026-04-15)

A full security audit was conducted by the Architecture Agent across 50+ files covering authentication, input validation, API security, data exposure, session management, secrets handling, upload security, and dependencies. The audit report lives at `docs/security-audit-2026-04-15.md`.

### Audit findings & remediations

| Severity | Found | Fixed | Remaining |
|----------|-------|-------|-----------|
| **Critical** | 2 | 2 | 0 |
| **High** | 5 | 5 | 0 |
| **Medium** | 6 | 2 | 4 (deferred, low external risk) |
| **Low** | 4 | 1 | 3 (deferred, acceptable at current scale) |

### Critical fixes

1. **Secrets exposure** — `.env.vercel.prod` contained ALL production API keys in plaintext and was not in `.gitignore`. Fixed: added to `.gitignore`, removed from tracking. **All keys must be rotated on provider dashboards.**

2. **Unauthenticated upload + GPT abuse** — `/api/micro-sessions` accepted anonymous file uploads to Vercel Blob and triggered OpenAI GPT-4o calls with no auth check. Fixed: added `getServerSession` gate before any file processing.

### High fixes

3. **Mobile token expiry** — Bearer tokens never expired, giving permanent access if stolen. Fixed: tokens now store `expires_at` (30 days) in the Account table. `getUserFromRequest()` rejects expired tokens. (`src/app/api/mobile/login/route.ts`, `src/lib/mobile-auth.ts`)

4. **Mobile login brute-force** — No rate limiting on `/api/mobile/login`. Fixed: IP-based rate limiter (10 attempts per 15 minutes). (`src/app/api/mobile/login/route.ts`)

5. **Cron routes publicly accessible** — All 7 cron routes used `if (secret && ...)` which skipped auth when `CRON_SECRET` was unset. Fixed: changed to `if (!secret || ...)` — missing secret now denies by default. (7 files in `src/app/api/cron/`)

6. **No upload validation** — All upload routes accepted arbitrary file types and sizes. Fixed: new `src/lib/upload-validator.ts` with presets:
   - `VIDEO_UPLOAD` — video/* only, max 50MB
   - `IMAGE_UPLOAD` — image/* only, max 10MB
   - `CMS_UPLOAD` — image/* + PDF, max 10MB
   - `SMALL_IMAGE_UPLOAD` — image/* only, max 5MB
   
   Applied to: `/api/sessions`, `/api/micro-sessions`, `/admin/cms/upload`, `/admin/arcade/upload-image`

7. **Debug endpoint** — `/api/debug/library-test` exposed stack traces without auth. Fixed: deleted entirely.

### Medium fixes

8. **Repeatable trial** — `POST /api/trial` allowed unlimited trial activations. Fixed: checks for past `trialStartedAt` before allowing new trial.

9. **Missing security headers** — Added to `next.config.mjs` for all non-embed routes:
   - `X-Frame-Options: SAMEORIGIN` (prevents clickjacking)
   - `Content-Security-Policy: frame-ancestors 'self'`
   - `X-Content-Type-Options: nosniff` (prevents MIME sniffing)
   - `Referrer-Policy: strict-origin-when-cross-origin`

### Low fix

10. **JWT session expiry** — NextAuth session had no `maxAge`. Fixed: set to 30 days. (`src/lib/auth.ts`)

### Deferred items (documented, lower priority)

| ID | Finding | Why deferred |
|---|---------|-------------|
| MED-001 | Wildcard CORS on some non-public routes | Needs per-route analysis; admin-only routes already auth-gated |
| MED-002 | `dangerouslySetInnerHTML` with unsanitized admin HTML | Admin-only content; external XSS risk is low |
| MED-003 | `/api/activity/track` and `/api/logs` unauthenticated | Design decision pending — anonymous tracking may be intentional |
| MED-006 | CORS on `/api/foundation` | Likely intentional for embed use case |
| LOW-002 | In-memory rate limiters reset per serverless instance | Acceptable at current scale; Redis upgrade planned for v2 |
| LOW-003 | PayPal plan slug from client params | Low-traffic endpoint with server-side amount validation |
| LOW-004 | All Blob uploads use `access: 'public'` | Needs signed URL infrastructure for private access |

### Security architecture patterns established

1. **Auth gate pattern**: Every non-public route MUST call `getServerSession(authOptions)` or `getUserFromRequest()` before any I/O (file upload, DB write, API call).

2. **Upload validation pattern**: Every file upload MUST call `validateUpload(file, PRESET)` before `put()`. Use the preset that matches the route's purpose.

3. **Cron auth pattern**: All cron routes MUST use `if (!secret || header !== ...)` — never `if (secret && ...)`. Missing secret = deny.

4. **Token lifecycle**: Mobile tokens expire after 30 days. Web sessions expire after 30 days. Both require re-authentication after expiry.

5. **Security headers**: Applied globally via `next.config.mjs`. Embed routes (`/embed/*`) are exempt (allow framing from any origin for viral distribution).
